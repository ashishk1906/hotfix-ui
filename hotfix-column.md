# Hotfix Dashboard Columns and Data Sources

This document explains what each dashboard column represents and where the data should come from in a live backend setup.

Core idea:

```txt
Hotfix creation data is stored in the backend database.
Each hotfix gets a unique job_id.
The UI uses that job_id to fetch the latest job details from backend APIs.
The frontend should not call GitHub, Jira, or CI tools directly.
```

## Column Mapping

| Column | Description | Example | Source of Truth | API Example | Feasibility & Notes |
|---|---|---|---|---|---|
| **Job ID** | Unique identifier for a hotfix job/run. | `CHF-329` | Backend job service. Generated when the hotfix is created. | `POST /api/v1/hotfixes` returns `{ "job_id": "CHF-329" }` | High. Can use UUID, sequence, or prefixed ID. |
| **Issue** | Ticket or issue linked to the hotfix. | `GH-329`, `JIRA-451` | Issue tracker reference stored during job creation. | `GET /api/v1/hotfixes/{job_id}` | High. Store only the issue key/reference in DB. |
| **Title** | Short description of the issue or fix. | `Checkout 500 on promo-code apply` | Jira issue summary or GitHub PR title, cached in backend. | `GET /api/v1/hotfixes/{job_id}` | High. Fetch once during creation and store it. |
| **Status** | Current lifecycle stage of the hotfix. | `building`, `waiting approval`, `completed`, `failed` | Backend orchestrator state stored in DB. CI/webhook events can update it. | `GET /api/v1/jobs/{job_id}` | High. Backend owns status. UI only displays it. |
| **Progress** | Visual progress percentage for the job. | `35%`, `80%`, `100%` | Backend mapping based on current status. | `GET /api/v1/jobs/{job_id}` returns `{ "progress_pct": 80 }` | High. Use a simple status-to-progress mapping instead of calculating real progress. |
| **Base** | Source branch where the fix came from. | `release/2026.05` | GitHub PR metadata, stored at job creation. | Backend may call `GET /repos/{owner}/{repo}/pulls/{pr_number}` | High. Snapshot once; do not call GitHub on every UI render. |
| **Target** | Destination hotfix branch. | `hotfix/CHF-329` | Generated by backend rules and stored in DB. | `GET /api/v1/hotfixes/{job_id}` | High. Usually created once during job setup. |
| **Commits** | Number of commits included in the hotfix. | `3` | GitHub PR/cherry-pick metadata cached in backend. | Backend may call `GET /repos/{owner}/{repo}/pulls/{pr_number}/commits` | Medium-high. Snapshot at creation; optionally refresh later. |
| **Risk** | Risk level for approval and prioritization. | `HIGH`, `MED`, `LOW` | Backend rules engine. | `GET /api/v1/jobs/{job_id}` | High. Can be calculated from files changed, commit count, area, severity, etc. |
| **Requester** | User who created the hotfix. | `arjun.mehta` | Authenticated user identity from SSO/JWT. | Captured during `POST /api/v1/hotfixes` | High. Store immutably at creation time. |
| **Approver** | Person or team responsible for approval. | `release-eng` | Internal approval workflow or GitHub review data. | `GET /api/v1/hotfixes/{job_id}` | High. Can be derived from branch rules, team ownership, or review events. |
| **Created** | Timestamp when the hotfix job was created. | `1h ago` | Backend DB `created_at` field. | `GET /api/v1/jobs/{job_id}` | High. Frontend formats timestamp as relative time. |
| **Updated** | Last time job data changed. | `6s ago` | Backend DB `updated_at` field. | `GET /api/v1/jobs/{job_id}` | High. Update whenever status, progress, approval, or logs change. |

## Recommended Backend APIs

Most dashboard data should come from the backend, not directly from external tools.

```http
POST /api/v1/hotfixes
GET  /api/v1/hotfixes
GET  /api/v1/hotfixes/{job_id}
GET  /api/v1/jobs/{job_id}
GET  /api/v1/jobs/{job_id}/logs
```

Recommended response shape:

```json
{
  "job_id": "CHF-329",
  "issue": "JIRA-451",
  "title": "Checkout 500 on promo-code apply",
  "status": "building",
  "progress_pct": 55,
  "base": "release/2026.05",
  "target": "hotfix/CHF-329",
  "commits": 3,
  "risk": "HIGH",
  "requester": "arjun.mehta",
  "approver": "release-eng",
  "created_at": "2026-05-25T10:20:00Z",
  "updated_at": "2026-05-25T10:26:00Z"
}
```

## How UI Gets Status Changes

The UI should read the latest status from the backend.

Simple approach:

```txt
Every few seconds:
UI calls GET /api/v1/hotfixes
Backend returns latest status/progress
UI re-renders the table
```

For example:

```txt
Running -> backend updates DB to Completed -> UI fetches again -> UI shows Completed
```

This can be done using polling first. Later, we can move to Server-Sent Events or WebSockets if we need real-time updates.

## Progress Column Options

### Option 1: Remove Progress

Use only the `Status` column.

This is the simplest option if the workflow stages are clear enough.

### Option 2: Backend Status-to-Progress Mapping

Backend maps each status to a progress percentage.

```json
{
  "created": { "progress": 5, "color": "gray" },
  "cherry picking": { "progress": 25, "color": "yellow" },
  "building": { "progress": 55, "color": "blue" },
  "waiting approval": { "progress": 80, "color": "purple" },
  "completed": { "progress": 100, "color": "green" },
  "failed": { "progress": 0, "color": "red" },
  "cancelled": { "progress": 0, "color": "gray" }
}
```

This is recommended because it is simple, predictable, and low cost.

### Option 3: Merge Status and Progress

Show a small progress bar inside or next to the status badge.

Example:

```txt
Status: building
Progress: 55%
```

This keeps the table compact and avoids an extra column.

## External Data Fetching Notes

### GitHub Data

Columns such as `Base`, `Target`, `Commits`, and sometimes `Approver` can come from GitHub.

Suggested GitHub calls:

```http
GET /repos/{owner}/{repo}/pulls/{pr_number}
GET /repos/{owner}/{repo}/pulls/{pr_number}/reviews
```

Use these to fetch:

```txt
base.ref
head.ref
commits
reviewer / approver
```

Important:

```txt
Call GitHub during job creation or refresh.
Cache the result in the backend DB.
Do not call GitHub every time the dashboard table renders.
```

### Jira or Issue Tracker Data

Columns such as `Issue` and `Title` can come from Jira or GitHub Issues.

Suggested Jira call:

```http
GET /rest/api/3/issue/{issueKey}
```

Use this to fetch:

```txt
key
fields.summary
priority
assignee
```

Important:

```txt
Store the issue key and title in the backend job record.
The frontend reads from our backend only.
```

## Recommended Approach

Start simple:

```txt
1. Store hotfix job data in backend DB.
2. Generate job_id on creation.
3. Cache GitHub/Jira metadata at creation time.
4. Let backend own status and progress.
5. Let CI/webhooks update backend status.
6. Let UI fetch from backend using GET /api/v1/hotfixes.
7. Use polling first, then SSE/WebSockets only if real-time updates are required.
```
