# Hotfix Table Columns and Data Sources

This document explains the columns currently shown in the Hotfix Dashboard table and where each value should come from in a live backend setup.

Core idea:

```txt
Hotfix data is stored in the backend database.
Each hotfix has a unique ID.
The UI fetches table data from backend APIs.
The frontend should not call GitHub, Jira, or CI/CD tools directly.
```

## Current UI Columns

| Column | Description | Example | Source of Truth | API Example | Feasibility & Notes |
|---|---|---|---|---|---|
| **ID** | Unique identifier for the hotfix job or servicing run. | `CHF-401` | Backend job service. Generated when the hotfix is created. | `POST /api/v1/hotfixes` returns `{ "id": "CHF-401" }` | High. Can use UUID, sequence, or prefixed ID. |
| **Issue** | Short issue/fix description shown in the table. | `Payment retry issue` | Jira issue summary, GitHub issue title, PR title, or value entered during hotfix creation. Stored in backend DB. | `GET /api/v1/hotfixes` | High. Fetch or enter once during creation and cache it. |
| **Branch** | Hotfix branch used for the fix. | `hotfix/CHF-401` | Generated by backend branch rules or selected during creation. Stored in backend DB. | `GET /api/v1/hotfixes` | High. Usually created once during job setup. |
| **Base** | Source/release branch the fix is based on. | `release/2026.05` | Git metadata from PR/base branch or manual selection during creation. Stored in backend DB. | `GET /api/v1/hotfixes` | High. Snapshot once; avoid repeated GitHub calls on every render. |
| **Status** | Current lifecycle state of the hotfix. | `running`, `waiting_approval`, `completed`, `failed` | Backend orchestrator state stored in DB. CI/CD webhook events can update it. | `GET /api/v1/hotfixes` or `GET /api/v1/hotfixes/{id}` | High. Backend owns status. UI only displays latest value. |
| **Risk** | Risk level used for approval and prioritization. | `high`, `medium`, `low` | Backend rules engine or manual selection during creation. Stored in backend DB. | `GET /api/v1/hotfixes` | High. Can be computed from commit count, files changed, service area, severity, etc. |
| **Owner** | Person responsible for the hotfix. | `Rahul Sharma` | Requester, assignee, or authenticated user captured during creation. Stored in backend DB. | `GET /api/v1/hotfixes` | High. Store user identity at creation time. |
| **Updated** | Last time the hotfix state changed. | `2m ago`, `just now` | Backend `updated_at` timestamp. Updated whenever status, risk, owner, branch, or workflow state changes. | `GET /api/v1/hotfixes` | High. Frontend can format timestamp as relative time. |

## Recommended Backend APIs

The dashboard table can be powered by one main list endpoint.

```http
POST /api/v1/hotfixes
GET  /api/v1/hotfixes
GET  /api/v1/hotfixes/{id}
```

Example response for `GET /api/v1/hotfixes`:

```json
{
  "hotfixes": [
    {
      "id": "CHF-401",
      "issue": "Payment retry issue",
      "branch": "hotfix/CHF-401",
      "base": "release/2026.05",
      "status": "running",
      "risk": "high",
      "owner": "Rahul Sharma",
      "updated_at": "2026-05-25T10:26:00Z"
    }
  ]
}
```

## How UI Knows Status Changed

The backend is the source of truth for status.

The UI reads the latest status from the backend.

Simple approach:

```txt
Every few seconds:
UI calls GET /api/v1/hotfixes
Backend returns latest hotfix rows
UI re-renders the table
```

Example:

```txt
Status changes in backend: running -> completed
UI fetches GET /api/v1/hotfixes again
Backend returns status: completed
UI updates the Status column
```

For the first version, polling every 5 seconds is enough. Later, if real-time updates are required, we can use Server-Sent Events or WebSockets.

## External Data Fetching Notes

### GitHub Data

Columns like `Branch` and `Base` can come from GitHub PR metadata.

Possible GitHub call:

```http
GET /repos/{owner}/{repo}/pulls/{pr_number}
```

Useful fields:

```txt
base.ref
head.ref
title
```

Important:

```txt
Call GitHub during hotfix creation or backend refresh.
Store the result in the backend DB.
Do not call GitHub directly from the frontend table.
Do not call GitHub every time the table renders.
```

### Jira or Issue Tracker Data

The `Issue` column can come from Jira, GitHub Issues, or manual input.

Possible Jira call:

```http
GET /rest/api/3/issue/{issueKey}
```

Useful fields:

```txt
key
fields.summary
priority
assignee
```

Important:

```txt
Store the issue summary in the backend job record.
The frontend reads the final table data from our backend only.
```

## Recommended Approach

```txt
1. Create hotfix using POST /api/v1/hotfixes.
2. Backend generates ID and branch.
3. Backend stores issue, branch, base, status, risk, owner, and updated_at.
4. CI/CD or backend workflow updates status.
5. UI calls GET /api/v1/hotfixes.
6. UI renders the table from backend response.
7. UI polls every few seconds to show status changes.
```
